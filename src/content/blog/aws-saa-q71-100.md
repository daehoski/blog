## Q71. DynamoDB 데이터 손상 대비 RPO 15분·RTO 1시간

**[상황]** 회사가 DynamoDB로 고객 정보를 저장합니다. 데이터 손상의 경우 **RPO 15분, RTO 1시간**을 충족해야 합니다.

**[요구사항]** RPO 15분·RTO 1시간을 충족하는 재해 복구 전략이 필요합니다.

**[보기]**
- A. DynamoDB 글로벌 테이블 구성 → 다른 리전으로 리디렉션
- B. **DynamoDB 지정 시간 복구(PITR) 구성 → 원하는 시점으로 복원**
- C. DynamoDB 데이터를 매일 S3 Glacier로 내보내기
- D. 15분마다 EBS 스냅샷 예약

**[정답]** 👉 **B - DynamoDB PITR (Point-In-Time Recovery)**

**[해설]** DynamoDB PITR은 지난 **35일 이내 임의의 시점**으로 데이터를 복원할 수 있습니다. 데이터 손상 발견 시 손상 이전 특정 순간(15분 전 등)으로 복원 가능하여 RPO 15분을 충족합니다. 글로벌 테이블(A)은 리전 장애 대응에는 좋으나 데이터 손상 시 모든 복제본에 전파되어 부적합합니다. Glacier(C)는 검색 시간이 길어 RTO 불충족입니다.

> 💡 **암기**: DynamoDB 특정 시점으로 복구 = **PITR (최대 35일)** / 리전 장애 대비 = **글로벌 테이블**

## Q72. EC2에서 S3 빈번한 업/다운로드 데이터 전송 비용 절감

**[상황]** 사진 처리 앱이 동일 리전 EC2와 S3 버킷 사이에서 사진을 **자주 업로드·다운로드**합니다. 데이터 전송 비용이 증가하고 있습니다.

**[요구사항]** EC2↔S3 데이터 전송 비용을 줄여야 합니다.

**[보기]**
- A. Amazon API Gateway를 퍼블릭 서브넷에 배포하고 라우팅 테이블 조정
- B. NAT 게이트웨이를 퍼블릭 서브넷에 배포하고 엔드포인트 정책 연결
- C. 앱을 퍼블릭 서브넷에 배포하고 인터넷 게이트웨이로 라우팅
- D. **S3 VPC 게이트웨이 엔드포인트를 VPC에 배포하고 S3 버킷 접근 허용 엔드포인트 정책 연결**

**[정답]** 👉 **D - S3 VPC 게이트웨이 엔드포인트**

**[해설]** S3 VPC 게이트웨이 엔드포인트는 VPC와 S3 간에 프라이빗 네트워크 연결을 제공합니다. 인터넷이나 NAT 게이트웨이를 통하지 않아 **데이터 전송 비용이 절감**됩니다(동일 리전 S3 접근은 엔드포인트 사용 시 무료). Q4·Q42와 동일한 핵심 패턴입니다.

## Q73. 배스천 호스트와 프라이빗 EC2 보안 그룹 설정 (2가지 선택)

**[상황]** 프라이빗 서브넷에 Linux 앱 EC2, 퍼블릭 서브넷에 Linux 배스천 호스트가 있습니다. **사내 네트워크에서 인터넷을 통해** 배스천 호스트에 연결하고, 배스천을 통해 앱 서버에 접근해야 합니다.

**[요구사항]** 보안 그룹을 올바르게 구성해야 합니다. (2가지 선택)

**[보기]**
- A. 배스천 SG: 앱 인스턴스의 인바운드 액세스만 허용
- B. 배스천 SG: 회사의 내부 IP 범위에서만 인바운드 허용
- C. **배스천 SG: 회사의 외부(공용) IP 범위에서만 인바운드 허용**
- D. **앱 SG: 배스천 호스트의 프라이빗 IP에서만 인바운드 SSH 허용**
- E. 앱 SG: 배스천 호스트의 공용 IP에서만 인바운드 SSH 허용

**[정답]** 👉 **C + D**

**[해설]** 트래픽 흐름: 사내 네트워크 → **인터넷** → 배스천 호스트(퍼블릭) → 앱 서버(프라이빗)
- **C**: 사내 네트워크에서 나갈 때는 외부(공용) IP로 나오므로, 배스천 호스트는 회사의 **외부 IP**에서의 인바운드를 허용
- **D**: 배스천에서 앱 서버로 들어올 때, VPC 내부이므로 배스천의 **프라이빗 IP**로 통신

## Q74. 2계층 웹 앱 보안 그룹 설정 (2가지 선택)

**[상황]** 2계층 웹 앱에서 퍼블릭 서브넷 EC2는 웹 서버, 프라이빗 서브넷 EC2는 Microsoft SQL Server DB를 실행합니다. 보안이 최우선입니다.

**[요구사항]** 보안 그룹을 어떻게 구성해야 하는지 선택해야 합니다. (2가지 선택)

**[보기]**
- A. **웹 계층 SG: 0.0.0.0/0에서 포트 443 인바운드 허용**
- B. 웹 계층 SG: 0.0.0.0/0에서 포트 443 아웃바운드 허용
- C. **DB 계층 SG: 웹 계층 SG에서 포트 1433 인바운드 허용**
- D. DB 계층 SG: 웹 계층으로 포트 443·1433 아웃바운드 허용
- E. DB 계층 SG: 웹 계층에서 포트 443·1433 인바운드 허용

**[정답]** 👉 **A + C**

**[해설]**
- **A**: 인터넷→웹 서버. 모든 인터넷(0.0.0.0/0)에서 HTTPS(443) 인바운드 허용
- **C**: 웹 서버→DB. 웹 계층 보안 그룹에서 SQL Server 포트(1433) 인바운드 허용

보안 그룹은 상태 유지(Stateful)라 인바운드 허용 시 응답 트래픽은 자동 허용됩니다. 아웃바운드 규칙은 별도로 설정할 필요가 없습니다.

## Q75. RESTful 다계층 앱 계층 오버로드 시 트랜잭션 손실 방지

**[상황]** 다계층 앱이 RESTful 서비스로 서로 통신합니다. **한 계층이 오버로드되면 트랜잭션이 삭제**됩니다. AWS로 이전하면서 현대화하려 합니다.

**[요구사항]** 트랜잭션 손실을 방지하고 운영 효율성이 가장 높은 솔루션이 필요합니다.

**[보기]**
- A. **API Gateway + Lambda 함수 + SQS를 앱 서비스 간 통신 계층으로 사용**
- B. CloudWatch로 성능 기록 분석 → EC2 인스턴스 크기 증가 (수직 확장)
- C. SNS로 EC2 Auto Scaling 그룹 간 메시징 + CloudWatch로 SNS 대기열 모니터링
- D. SQS로 EC2 Auto Scaling 그룹 간 메시징 + CloudWatch로 SQS 대기열 모니터링 후 확장

**[정답]** 👉 **A - API Gateway + Lambda + SQS**

**[해설]** RESTful API = API Gateway, 계층 간 트랜잭션 손실 방지 = SQS. API Gateway가 요청을 받아 Lambda로 전달하고, 계층 간에 SQS가 버퍼 역할을 합니다. 한 계층이 오버로드되어도 SQS가 트랜잭션을 안전하게 보관합니다. 완전 서버리스 아키텍처로 운영 효율이 가장 높습니다.

## Q76. 온프레미스 공장 데이터(JSON) S3에 안전하게 전송

**[상황]** 단일 공장의 여러 기계에서 매일 **10TB의 JSON 계측 데이터**를 생성합니다. SAN에 저장된 데이터를 **실시간에 가까운 분석**을 위해 S3로 전송해야 합니다. **안전한 전송**이 중요합니다.

**[요구사항]** 가장 안정적으로 온프레미스→S3 데이터를 안전하게 전송해야 합니다.

**[보기]**
- A. 공용 인터넷을 통한 AWS DataSync
- B. **AWS Direct Connect를 통한 AWS DataSync**
- C. 공용 인터넷을 통한 AWS DMS
- D. Direct Connect를 통한 AWS DMS

**[정답]** 👉 **B - Direct Connect + AWS DataSync**

**[해설]**
- **DataSync** vs **DMS**: DataSync는 파일/객체 스토리지 데이터 전송용, DMS는 DB 마이그레이션용. JSON 파일 S3 전송에는 DataSync가 적합합니다.
- **Direct Connect**: 전용선으로 인터넷을 통하지 않아 안전하고 안정적. DataSync는 자동 암호화·무결성 검증을 포함합니다.
- 공용 인터넷(A·C)은 노출 위험이 있어 "안전한 전송" 요건 불충족입니다.

## Q77. 실시간 데이터 수집 API + 변환 + S3 저장 (최소 운영 오버헤드)

**[상황]** 회사가 **실시간 데이터 수집 아키텍처**를 구성해야 합니다. 스트리밍되는 데이터를 변환하는 프로세스인 **API와 스토리지 솔루션**이 필요합니다.

**[요구사항]** 최소 운영 오버헤드로 API→실시간 수집→변환→S3 저장 파이프라인을 구축해야 합니다.

**[보기]**
- A. EC2에 API 호스팅 → Kinesis Data Streams → Kinesis Firehose → Lambda 변환 → S3
- B. EC2에 API 호스팅 → AWS Glue → S3
- C. **API Gateway API → Kinesis Data Streams → Kinesis Data Firehose → Lambda 변환 → S3**
- D. API Gateway API → AWS Glue → Lambda 변환 → S3

**[정답]** 👉 **C - API Gateway + Kinesis Streams + Firehose + Lambda + S3**

**[해설]** 최소 운영 오버헤드를 위해 API는 EC2 대신 **API Gateway(완전관리형)**를 사용합니다. API Gateway를 Kinesis Data Streams와 통합하여 데이터를 수집하고, Firehose가 Streams에서 데이터를 받아 Lambda로 변환 후 S3에 전달합니다. 완전 서버리스 파이프라인입니다.

## Q78. DynamoDB 데이터 7년 보관 (가장 운영 효율적)

**[상황]** 회사가 사용자 트랜잭션 데이터를 DynamoDB에 보관해야 합니다. 데이터를 **7년간 보관**해야 합니다.

**[요구사항]** 가장 운영 효율적인 방법으로 7년 보관 솔루션을 구현해야 합니다.

**[보기]**
- A. DynamoDB PITR로 지속 백업
- B. **AWS Backup으로 테이블에 대한 백업 일정·보존 정책 생성**
- C. DynamoDB 콘솔에서 주문형 백업 → S3 버킷에 저장 → S3 수명 주기 구성
- D. EventBridge → Lambda → 테이블 백업 → S3 저장 → S3 수명 주기

**[정답]** 👉 **B - AWS Backup**

**[해설]** PITR(A)는 최대 35일만 지원하여 7년 요건 불충족입니다. **AWS Backup**은 한 곳에서 DynamoDB 백업을 예약·복사·태그 지정·수명 주기 관리가 가능하며, 콜드 스토리지로 전환하여 7년 간 비용 효율적으로 유지합니다. C·D는 수동 또는 추가 Lambda 유지보수가 필요하여 운영 오버헤드가 큽니다.

## Q79. 사용량 예측 불가 DynamoDB 비용 최적화

**[상황]** 회사가 DynamoDB 사용을 계획합니다. **아침에는 거의 사용하지 않고**, **저녁에는 트래픽이 예측 불가능**하며 급증이 매우 빠르게 발생합니다. 비용 최적화가 우선입니다.

**[요구사항]** 예측 불가능한 워크로드에 비용 효율적인 DynamoDB 모드를 선택해야 합니다.

**[보기]**
- A. **온디맨드 용량 모드(On-Demand Capacity Mode)로 DynamoDB 테이블 생성**
- B. 글로벌 보조 인덱스가 있는 DynamoDB 테이블 생성
- C. 프로비저닝된 용량 + Auto Scaling으로 DynamoDB 테이블 생성
- D. 프로비저닝된 용량 + 글로벌 테이블 구성

**[정답]** 👉 **A - DynamoDB 온디맨드 용량 모드**

**[해설]** 온디맨드 모드는 **사용한 만큼만 과금**합니다. 트래픽 급증 시 DynamoDB가 자동으로 즉시 용량을 조절합니다. 프로비저닝 모드(C·D)는 예상 용량을 사전에 설정해야 하는데, 사용량이 예측 불가능하면 낭비가 발생합니다.

> 💡 **암기**: 트래픽 예측 불가/불규칙 = **DynamoDB 온디맨드** / 예측 가능·지속적 = **프로비저닝+AutoScaling**

## Q80. KMS 암호화된 AMI를 파트너 계정에 안전하게 공유

**[상황]** 기존 AWS 계정의 AMI를 **MSP 파트너의 AWS 계정과 공유**해야 합니다. AMI는 EBS 스냅샷을 **KMS 고객 관리형 키**로 암호화합니다.

**[요구사항]** 가장 안전한 방법으로 AMI를 파트너 계정과 공유해야 합니다.

**[보기]**
- A. 암호화된 AMI·스냅샷을 공개 설정 + 파트너 계정이 키를 사용하도록 키 정책 수정
- B. **AMI의 launchPermission 속성 수정하여 파트너 계정에만 공유 + 파트너 계정이 키를 사용하도록 기존 키 정책 수정**
- C. AMI launchPermission 수정 + 파트너가 소유한 새 KMS 키를 신뢰하도록 키 정책 수정
- D. AMI를 파트너 S3 버킷으로 내보내기 + 파트너 KMS 키로 암호화

**[정답]** 👉 **B - launchPermission + 기존 KMS 키 정책 수정**

**[해설]**
- `launchPermission` 속성으로 **특정 계정에만** AMI 사용 권한 부여 (공개 아님)
- 스냅샷 암호화에 사용된 KMS 키의 **키 정책**에 파트너 계정 추가
- C(파트너 키 신뢰)는 파트너가 키를 악용할 수 있어 보안 위험, A(공개)는 보안 요건 위반입니다.

## Q81. 병렬 처리 워크로드 느슨한 연결 + 영구 저장 아키텍처

**[상황]** 처리할 작업 수에 따라 앱 노드를 추가·제거하면서 **병렬로 실행**되어야 합니다. 프로세서는 **상태 비저장(stateless)**이며, **느슨하게 연결**되어 있고 **작업 항목이 영구 저장**되어야 합니다.

**[요구사항]** 느슨한 연결 + 영구 저장 + 작업 수 기반 자동 확장이 필요합니다.

**[보기]**
- A. SNS 주제 + AMI 기반 시작 구성 + Auto Scaling 그룹 (CPU 사용량 기반 확장)
- B. SQS 대기열 + AMI 기반 시작 구성 + Auto Scaling 그룹 (네트워크 사용량 기반 확장)
- C. **SQS 대기열 + AMI 기반 시작 템플릿 + Auto Scaling 그룹 (SQS 대기열 항목 수 기반 확장)**
- D. SNS 주제 + AMI 기반 시작 템플릿 + Auto Scaling 그룹 (SNS 메시지 수 기반 확장)

**[정답]** 👉 **C - SQS + 시작 템플릿 + SQS 항목 수 기반 Auto Scaling**

**[해설]**
- **SQS**: 작업 항목을 영구적으로 저장, 느슨한 연결 제공 (SNS는 영구 저장 불가)
- **시작 템플릿**: 시작 구성보다 더 현대적인 AWS 권장 방식
- **SQS 항목 수 기반 확장**: 처리 대기 작업 수에 가장 직접적으로 대응 (CPU·네트워크보다 명확)

## Q82. ACM으로 가져온(Import) 인증서 만료 30일 전 알림

**[상황]** 회사가 ACM으로 **가져온 인증서**를 ELB에 사용합니다. 각 인증서가 만료되기 **30일 전에 보안팀에 알려야** 합니다.

**[요구사항]** 인증서 만료 30일 전 자동 알림을 구현해야 합니다.

**[보기]**
- A. ACM에 규칙 추가 → 만료 30일 전부터 매일 SNS에 메시지 게시
- B. **AWS Config 규칙으로 30일 이내 만료 인증서 확인 → EventBridge가 비준수 리소스 보고 시 SNS 알림 호출**

- D. EventBridge 규칙으로 30일 이내 만료 인증서 감지 → Lambda → SNS 알림

**[정답]** 👉 **B - AWS Config + EventBridge + SNS**

**[해설]** AWS Config는 가져온(Imported) ACM 인증서의 만료 날짜를 모니터링하는 관리형 규칙을 제공합니다. 비준수 상태(만료 임박) 감지 시 EventBridge가 이를 포착하여 SNS로 알림을 발송합니다. A처럼 매일 알림하는 것이 아니라 30일 전 한 번 알립니다. D(EventBridge)는 이벤트 발생 이후에 반응하는 서비스로 미래 날짜 감시에는 Config가 더 적합합니다.

## Q83. 온프레미스 동적 웹사이트 유럽 사용자에게 빠르게 제공

## Q85. 업로드된 문서 수정·삭제 불가 (규제 요구사항)

**[상황]** 프로덕션 웹 앱에서 사용자가 문서를 업로드합니다. **새 규제 요구사항**에 따라 저장 후 **문서를 수정하거나 삭제할 수 없습니다**.

**[요구사항]** 저장된 문서의 수정·삭제를 방지해야 합니다.

**[보기]**
- A. **S3 버전 관리 + S3 객체 잠금(Object Lock)이 활성화된 S3 버킷에 저장**
- B. S3 버킷에 저장 + S3 수명 주기 정책으로 주기적 보관
- C. S3 버전 관리 활성화 + ACL로 읽기 전용 제한
- D. Amazon EFS 볼륨에 저장 + 읽기 전용 모드로 마운트

**[정답]** 👉 **A - S3 버전 관리 + S3 Object Lock**

**[해설]** **S3 Object Lock**은 WORM(Write Once Read Many) 모델로 객체의 삭제·덮어쓰기를 방지합니다. 버전 관리와 함께 사용하면 각 버전에 대해 잠금을 적용할 수 있습니다. ACL(C)은 접근 제어이지 잠금이 아니며, EFS(D)를 읽기 전용으로 마운트하면 업로드 자체가 불가합니다.

## Q86. RDS MySQL 자격증명 안전하게 관리 + 자주 교체

**[요구사항]** 자격증명 빈번한 교체 + 안전한 저장·관리가 필요합니다.

**[보기]**

## Q87. DB 업그레이드 중 Lambda가 받은 고객 데이터 보존

**[상황]** API Gateway + Lambda → Aurora MySQL 아키텍처에서 **DB 업그레이드 중** Lambda가 DB 연결을 설정하지 못합니다. 그 결과 **일부 이벤트의 고객 데이터가 기록되지 않습니다**.

**[요구사항]** DB 업그레이드 중 생성되는 고객 데이터를 안전하게 저장해야 합니다.

**[보기]**
- A. Lambda와 DB 사이에 RDS Proxy 프로비저닝
- B. Lambda 실행 시간 최대화 + 데이터 저장 코드에 재시도 메커니즘 추가
- C. 고객 데이터를 Lambda 로컬 스토리지에 유지 → 새 Lambda로 스캔·저장
- D. **SQS FIFO 대기열에 고객 데이터 저장 → 새 Lambda 함수가 대기열을 폴링하여 DB에 저장**

**[정답]** 👉 **D - SQS FIFO + 별도 Lambda**

**[해설]** DB 업그레이드 중 데이터를 잃지 않으려면 임시 버퍼가 필요합니다. **SQS FIFO**에 데이터를 저장하면 DB 업그레이드 중에도 데이터가 안전하게 보관됩니다. 업그레이드 완료 후 새 Lambda가 SQS에서 순서대로 데이터를 꺼내 DB에 저장합니다. RDS Proxy(A)는 연결 풀링을 최적화하지만 DB가 완전히 다운된 경우에는 데이터를 보관하지 않습니다.
